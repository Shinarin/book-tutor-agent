# 拆解

输入一本书，只输出**一张主次重点表**——从头到尾，哪几章是重点、哪几章可以合并压缩、哪几章可以一笔带过。

## 这份产物给谁、做什么用

给**讲解者**看的一张行军图：

- 决定每一章用多少篇幅讲、按什么顺序铺
- 决定哪些章可以并成一节快讲完、哪些章可以一句话带过
- 不至于从第一章平铺到最后一章、把作者随手的展开和核心论断当成同等重量讲

## 输入材料

- **全书核心知识点**：`{keypoints_file}`。
- **skeleton**：`{skeleton_path}`。原书的索引——每章一个 skeleton 文件，每个条目带原书 `L{start}-L{end}` 行号区间。**先读 `{skeleton_path}/index.md` 拿到章节次序**，再按需挑具体 skeleton 文件读；不要靠 `Glob`/`LS` 的返回顺序判断章节先后。
- **原书**：`{book_path}`。**不要通读**。只在某章定档拿不准、需要回去看作者怎么写时再读，先用 skeleton 拿到行号区间，再用 `Read` 按 `offset`/`limit` 精读那几段。

## 怎么拆

### 定档

照原书章节次序走一遍。每一章定一档，三档：

- **重点**——支撑全书主轴的章。少了它主轴立不住。讲解时要拿出最多篇幅、最仔细铺。
- **可压缩**——作者展开、补充、应对反驳、给例子的章。骨架要讲，细节可略。**多个相邻的可压缩章常常能合并成一节讲**。
- **可略**——作者顺手推到别处的应用、引申、题外话、案例堆叠。讲解时一两句话带过，或干脆跳过。

定档的判断问句，对着每一章问：

- 拿掉这一章，全书主轴还立得住吗？立不住 → 重点。
- 拿掉这一章，主轴还立得住、但论证会变薄？→ 可压缩。
- 拿掉这一章，主轴不受影响？→ 可略。
- 这一章的内容是不是已经过时（数据陈旧、技术/工具被取代、行业格局已变、结论被后续研究推翻）？是 → 可略，讲解时一句话点明"此处原书已过时"即可。

### 合并

定完档之后，从头到尾再扫一遍，找相邻的可压缩章：如果讲的是同一件事的不同侧面（例如都是同一论点的不同案例、都是同一方法的不同应用场景），写明"第 X 章 + 第 Y 章可合并成一节，讲 ……"。

## 输出

写入 `{output_path}`（覆盖写入）。Markdown 格式。表格、列表、分段都可以——挑最能让讲解者一眼看清主次、合并、压缩三件事的形式。

### 写完自检

- 标"重点"的章加起来，能不能撑住全书主轴？撑不住 → 漏了重点章，回去重定。
- 标"重点"的章是不是占了全书一半以上？占了 → 没敢下手，主次不分明，回去再压。
- 合并建议是不是真的能合？把相邻两章的实际内容拉出来对一下，确实是同一件事的两面，才能合。
